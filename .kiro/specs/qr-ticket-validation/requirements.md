# Requirements Document

## Introduction

The QR Ticket Validation feature adds a scannable QR code to every PDF ticket generated by the HyperCore Gift Draw Platform. The QR code encodes a public validation URL containing the ticket's UUID. A public, unauthenticated endpoint allows anyone — event staff, ticket holders, or attendees — to scan the code and immediately verify whether a ticket is active or cancelled. This is especially important after cancellations and folio reassignments, where a physical or printed ticket may no longer be valid.

The feature builds on the existing `Ticket` model (which already uses a UUID primary key), the existing reportlab PDF generator, and the existing `PublicEndpointThrottle`. No database schema changes are required.

## Glossary

- **QR_Generator**: The `generate_qr_image()` function in `backend/core/qr_utils.py` responsible for producing QR code PNG bytes.
- **PDF_Generator**: The `generate_ticket_pdf()` function in `backend/core/ticket_generator.py` responsible for producing PDF ticket bytes.
- **Validation_Endpoint**: The public Django REST Framework view `TicketValidateView` at `GET /api/tickets/{uuid}/validate`.
- **Ticket**: A Django model instance representing a raffle ticket with a UUID primary key, folio, buyer name, phone, and status (`active` or `cancelled`).
- **Validation_URL**: The URL encoded in the QR code, formatted as `{SITE_BASE_URL}/api/tickets/{ticket.id}/validate`.
- **SITE_BASE_URL**: A Django setting providing the public-facing base URL of the API (e.g., `https://api.hypercoreqro.lat`).
- **PublicEndpointThrottle**: The existing DRF throttle class limiting unauthenticated requests to 30 per minute.
- **Folio**: The human-readable ticket identifier (e.g., `HC-005`). Multiple `Ticket` rows may share a folio (one active, others cancelled).

## Requirements

### Requirement 1: QR Code Generation

**User Story:** As a ticket buyer, I want a QR code on my PDF ticket, so that I can quickly verify my ticket's validity by scanning it at the event.

#### Acceptance Criteria

1. THE QR_Generator SHALL construct the Validation_URL as `{base_url}/api/tickets/{ticket_id}/validate` for any given ticket UUID and base URL.
2. WHEN a valid ticket UUID and a non-empty base URL are provided, THE QR_Generator SHALL return non-empty bytes representing a valid PNG image.
3. WHEN the PNG bytes returned by THE QR_Generator are decoded as a QR code, THE decoded string SHALL equal exactly `{base_url}/api/tickets/{ticket_id}/validate` with no extra characters or truncation.
4. THE QR_Generator SHALL use ERROR_CORRECT_M error correction level to allow recovery from approximately 15% damage on printed tickets.
5. THE QR_Generator SHALL produce a QR image sized appropriately for embedding at approximately 22mm × 22mm on an A6 PDF without loss of scannability.

### Requirement 2: Public Ticket Validation Endpoint

**User Story:** As an event staff member, I want to scan a ticket's QR code and see its current status, so that I can confirm whether a ticket is valid before granting entry or awarding a prize.

#### Acceptance Criteria

1. WHEN a GET request is made to `/api/tickets/{uuid}/validate` with a UUID that matches an existing ticket, THE Validation_Endpoint SHALL return HTTP 200 with a JSON body containing exactly the fields `id`, `folio`, `status`, and `full_name`.
2. WHEN a GET request is made to `/api/tickets/{uuid}/validate` with a UUID that does not match any ticket, THE Validation_Endpoint SHALL return HTTP 404 with body `{"detail": "Ticket not found."}`.
3. THE Validation_Endpoint SHALL accept requests without any authentication token or session credential.
4. THE Validation_Endpoint SHALL never include `phone`, `created_by`, `created_at`, or `cancelled_at` in any response body.
5. WHEN a client sends more than 30 requests per minute to the Validation_Endpoint, THE Validation_Endpoint SHALL return HTTP 429 Too Many Requests.
6. WHEN a non-UUID string is provided in the `{uuid}` path segment, THE Validation_Endpoint SHALL return HTTP 404 without reaching the view logic.
7. THE Validation_Endpoint SHALL perform no write operations on any data.

### Requirement 3: QR Code Embedding in PDF Tickets

**User Story:** As an admin, I want every downloaded PDF ticket to include a QR code, so that ticket holders automatically receive a scannable validation code without any extra steps.

#### Acceptance Criteria

1. WHEN `generate_ticket_pdf()` is called for any ticket, THE PDF_Generator SHALL invoke THE QR_Generator and embed the resulting QR image in the output PDF.
2. WHEN `generate_ticket_pdf()` is called with an empty `base_url` argument, THE PDF_Generator SHALL use the value of `SITE_BASE_URL` from Django settings as the base URL for the Validation_URL.
3. THE PDF_Generator SHALL position the QR code in the lower-right area of the A6 ticket, above the footer, without overlapping the folio number, participant card, or date chip.
4. THE PDF_Generator SHALL render a label `"ESCANEA PARA VALIDAR"` below the QR code image.
5. THE PDF_Generator SHALL preserve all existing ticket elements — header, brand name, folio, participant card, date chip, and footer — when embedding the QR code.
6. WHEN `generate_ticket_pdf()` is called, THE PDF_Generator SHALL return non-empty bytes representing a valid PDF document.

### Requirement 4: Cancellation and Reassignment Isolation

**User Story:** As an event organizer, I want cancelled tickets to always show as cancelled when scanned, even after their folio is reassigned to a new buyer, so that fraudulent use of old printed tickets is prevented.

#### Acceptance Criteria

1. WHEN a ticket's status is set to `cancelled`, THE Validation_Endpoint SHALL return `"status": "cancelled"` for that ticket's UUID on all subsequent requests.
2. WHEN a folio is reassigned via the reassign endpoint, THE System SHALL create a new Ticket record with a UUID different from the original cancelled ticket's UUID.
3. WHEN a folio is reassigned, THE Validation_Endpoint SHALL continue to return `"status": "cancelled"` for the original ticket's UUID.
4. WHEN a folio is reassigned, THE Validation_Endpoint SHALL return `"status": "active"` for the new ticket's UUID.

### Requirement 5: Configuration and Dependencies

**User Story:** As a backend developer, I want the QR validation feature to be properly configured and packaged, so that it works correctly in both local development and production environments.

#### Acceptance Criteria

1. THE System SHALL include `qrcode[pil]>=7.4` in `backend/requirements.txt`.
2. THE System SHALL define a `SITE_BASE_URL` Django setting that reads from the `SITE_BASE_URL` environment variable with a default value of `http://localhost:8000`.
3. THE System SHALL expose the Validation_Endpoint at the URL path `api/tickets/<uuid:ticket_id>/validate` in the Django URL configuration.
4. THE QR_Generator SHALL be implemented as a standalone function in `backend/core/qr_utils.py` with no side effects.
